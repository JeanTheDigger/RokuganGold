## 57.1 Allowlist Model --- Scoring Architecture Change --- LOCKED

SUPERSEDES: The schema note in Section 55.14 and Annex A (objective_alignment.json) previously stated "Missing entries default to 0." This is replaced with: "Missing entries are BLOCKED --- only listed actions are scored for each NeedType."

**Rationale: **Alignment score compression (Section 57.2) narrows the gap between top-cluster actions from ~30 points to ~10 points. Under the old "missing = 0" rule, unlisted actions could accumulate +115 from other scoring components (personality lean +25, competence +20, disposition +25, urgency +30, standing +15, topic +15) and outscore listed actions. Stress testing confirmed absurd outcomes: INTIMIDATE beating ASK_FOR_INTRODUCTION for IDENTIFY_CONTACT, ORDER_BATTLE beating PROBE for GATHER_INTELLIGENCE, PERFORM_FOR beating ORDER_FORTIFY for DEFEND_PROVINCE. The allowlist model prevents unlisted actions from entering the scoring pool, making compression safe.

**Implementation: **In Phase 4 (or early Phase 5 before scoring), if an ActionID has no entry in objective_alignment.json for the current NeedType, remove it from the option pool. One condition check. DO_NOTHING is listed at 0 in every NeedType entry, so the deadlock fallback always survives. The Phase 6 deadlock path (Section 55.4.6) is unaffected --- it bypasses alignment scoring entirely.

**Extension contract update: **Section 55.16.1 Step 3 becomes mandatory: "Add objective_alignment.json entries for every NeedType where this action should be available. Actions not listed for a NeedType are blocked for that NeedType." Section 55.16.6 completeness test must verify the action appears in at least one ScoredAction.

**Failure mode: **If a developer forgets to list a legitimate action for a NeedType, the NPC selects the next best available action --- suboptimal but sane. Under the old model, forgetting to block an absurd action produced visibly insane NPC behavior. The allowlist failure mode is strictly better.
