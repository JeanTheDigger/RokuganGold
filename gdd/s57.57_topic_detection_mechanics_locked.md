# Section 57.57 — Topic Detection Mechanics — LOCKED

## Overview

Four passive detection triggers fire on seasonal ticks (or daily, where noted)
to surface information that the simulation already holds but has not yet
propagated to the relevant observers. All four produce topics that feed the
standard topic momentum system. None create CrimeRecords — they surface
information, not new criminal acts.

Values in this section supersede the PROVISIONAL annotations in s14 (v617
note) and in the fugitive extradition system.

---

## 57.57.1 T4-48 FUGITIVE_SIGHTING

### Trigger Conditions

A `"fugitive_sighting"` topic fires whenever one of three deterministic paths
is satisfied. Each path is evaluated independently; the first matching path
fires. One topic per fugitive per season (dedup: one sighting topic per
`(fugitive_id, season)` pair, keyed on slug).

**Path A — Co-location at same settlement**

Fires daily. If a non-fugitive living character is co-located at the same
settlement as the fugitive (`physical_location` match, neither traveling),
a sighting topic is generated. The observer must not be the fugitive's
direct lord (the lord already knows their vassal's location).

**Path B — ORDER_PATROL active in province**

Fires at seasonal tick. If `patrolled_provinces` in `season_meta` contains
the province ID matching the fugitive's current settlement, a sighting topic
is generated. Patrol sightings represent the increased scrutiny of an active
patrol order.

**Path C — Fugitive Status ≥ 5.0 at time of T3-12 FUGITIVE status
declaration**

Fires at seasonal tick during the extradition seasonal pass. A fugitive of
Status ≥ 5.0 is notable enough that their departure from legitimate society
generates immediate public notice. The sighting is the social echo of the
declaration itself.

*Note: The existing code used threshold 3.0 for Path C. Per this section
(s57.57), the correct threshold is 5.0. Status 3.0–4.9 fugitives are still
visible via Path A and Path B.*

### Topic Fields

| Field | Value |
|-------|-------|
| `topic_type` | `"fugitive_sighting"` |
| `tier` | TIER_4 |
| `category` | POLITICAL |
| `subject_character_id` | fugitive's `character_id` |
| `clan_involved` | fugitive's `clan` |

### Listener Effects

Characters who receive this topic as an Opportunity Scanner candidate may
generate `ORDER_PATROL` (NeedType: INVESTIGATE_THREAT) or
`INVESTIGATE_THREAT` needs. No mechanical consequence beyond topic
propagation is defined in this section.

## 57.57.2 T4-13 DISAPPEARANCE

### Trigger Conditions

Fires at seasonal tick. All five conditions must be met:

1. Character is alive (`not CharacterStats.is_dead(c)`)
2. Character has not been seen socially for ≥ 1 IC season (90 IC days):
   `ic_day − c.last_social_ic_day >= 90` where `last_social_ic_day`
   is updated whenever the character participates in a daily conversation,
   sends a letter, or receives a letter.
3. No outgoing letter sent in the same window: no `LetterData` with
   `sender_id == c.character_id` and `ic_day_sent >= ic_day − 90` among
   `pending_letters`.
4. Character is not a fugitive: `c.legal_status != Enums.LegalStatus.FUGITIVE`.
5. Character is not legitimately traveling: `not TravelSystem.is_traveling(c)`.

Additionally, the targeted exclusion rule applies: characters who hold a
`CrimeRecord` with `perpetrator_id == c.character_id` and
`legal_status == Enums.LegalStatus.UNDER_INVESTIGATION` are excluded from
the holder list (a perpetrator who made someone disappear should not
receive the "where did they go?" topic).

### Holder Generation

Two paths generate holders for the disappearance topic:

- **Lord notification:** The character's `lord_id` holder receives the
  topic unconditionally (if `lord_id >= 0` and lord is alive).
- **Close-associate notification:** Characters at the last-seen settlement
  (matching `c.physical_location`) with disposition ≥ +31 (Friend tier per
  s12.2) toward the missing character receive the topic, subject to the
  exclusion rule above.

The topic is added to `active_topics` once. Holder seeding follows the
standard per-character topic-pool distribution path.

### Deduplication

One disappearance topic per character per season. Skip if an unresolved
`"disappearance"` topic with matching `subject_character_id` already exists
in `active_topics`.

### Topic Fields

| Field | Value |
|-------|-------|
| `topic_type` | `"disappearance"` |
| `tier` | TIER_4 |
| `category` | PERSONAL |
| `subject_character_id` | missing character's `character_id` |
| `clan_involved` | missing character's `clan` |

## 57.57.3 T3-39 JUSTICE_REFUSAL

### Trigger Conditions

Fires at seasonal tick. All five conditions must be met:

1. A `CrimeRecord` exists with `legal_status == Enums.LegalStatus.FUGITIVE`.
2. The crime's province (mapped from `record.location`) belongs to a
   different clan than the fugitive's current settlement province
   (cross-clan case only — same-clan fugitives are handled by internal
   clan justice).
3. An `extradition_request` topic exists in `active_topics` with
   `subject_character_id == record.perpetrator_id` (T4-78 prerequisite —
   a formal demand must have been issued before refusal can be declared).
4. The harboring clan lord has not issued a COOPERATE response in the same
   or previous season (evaluated by absence of a `"extradition_cooperation"`
   result in `crime_records` for the same perpetrator).
5. At least 90 IC days have elapsed since the extradition request topic was
   created (`ic_day − extradition_topic.ic_day_created >= 90`).

### Topic Fields

| Field | Value |
|-------|-------|
| `topic_type` | `"justice_refusal"` |
| `tier` | TIER_3 |
| `category` | POLITICAL |
| `subject_character_id` | fugitive's `character_id` |
| `clan_involved` | harboring clan |

### Escalation Chain

The formal escalation chain documented here for reference:
`T3-10 / T3-11 (crime conviction topic)` → `T4-78 (extradition_request)` →
`T3-39 (justice_refusal, this section)` → `T3-17 (imperial_intervention)`.

Imperial intervention (T3-17) is not generated by this section; it is a
downstream consequence handled by the Emperor's strategic review (s55.10).

### Deduplication

One justice_refusal topic per perpetrator per season. Skip if an unresolved
`"justice_refusal"` topic with matching `subject_character_id` already exists
in `active_topics`.

## 57.57.4 T4-90 BEHAVIORAL_ANOMALY

### Trigger Conditions

Fires at seasonal tick. The following path is implementable now:

**Path B — Correspondence gap**

A character who previously sent ≥ 3 letters (across all time) has sent no
letter in ≥ 90 IC days. Formally: `pending_letters` contains ≥ 3 letters
with `sender_id == c.character_id` total (any delivery state), and no letter
with `sender_id == c.character_id` has `ic_day_sent >= ic_day − 90`.

The 3-letter threshold establishes a pattern of correspondence before silence
becomes anomalous. A character who never wrote letters does not generate an
anomaly by continuing not to write.

**Path A — Financial audit discrepancy**

Deferred. Requires an audit system not currently implemented. Do not
implement until an audit mechanic is specified.

**Path C — Location anomaly (T4-91 sighting inconsistent with objective)**

Deferred. Requires T4-91 location_sighting topic type, which is not yet
defined. Do not implement until T4-91 is specified.

### Deduplication

One behavioral_anomaly topic per character per season. Skip if an unresolved
`"behavioral_anomaly"` topic with matching `subject_character_id` already
exists in `active_topics`.

### Topic Fields

| Field | Value |
|-------|-------|
| `topic_type` | `"behavioral_anomaly"` |
| `tier` | TIER_4 |
| `category` | PERSONAL |
| `subject_character_id` | subject character's `character_id` |
| `clan_involved` | subject character's `clan` |

### Holder Generation

The character's `lord_id` receives the topic (if `lord_id >= 0` and alive).
Close associates at the last-known settlement (same `physical_location`,
disposition ≥ +31) also receive it, same as DISAPPEARANCE.

## 57.57.6 Dropped Mechanic

**PUBLIC_MISCONDUCT_PATTERN** — Dropped as redundant with T3-40
`HABITUAL_VIOLENCE`, which already triggers on three or more
`DISHONORABLE_CONDUCT` CrimeRecords for the same perpetrator within a
rolling 4-season window. No additional mechanic is needed.

## 57.57.7 Deferred

- T4-90 Path A (financial audit discrepancy): blocked on audit system design.
- T4-90 Path C (location anomaly): blocked on T4-91 topic type definition.
- Listener effects beyond topic propagation for any of the four types.
- Kolat sleeper activation anomaly detection: handled through existing
  paths (financial and correspondence gaps produced by sleeper behavior
  feed directly into Paths A and B above). No separate sleeper-detection
  mechanic is specified.
