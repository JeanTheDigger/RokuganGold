## 57.45a Geisha Intelligence System — Numeric Values Lock

This addendum formalizes all PROVISIONAL numeric values for the Geisha Intelligence
System (Section 57.45). The mechanics are specified in s57.45. This document locks
the formulas and constants needed for implementation.

---

### A1–A3: Koku Costs per Geisha House Visit (§57.45.3, §57.44.5)

Calibrated against: Local Daimyo stipend = 1 koku/season (§4.3). Incense Ceremony
("genuinely steep") = the ceiling above Tier 3. Pleasure Quarter is below Tier 1.

| Constant | Value | Settlement Context |
|---|---|---|
| GEISHA_KOKU_TIER_1 | 0.1 koku | Provincial House — any samurai can afford occasional visits |
| GEISHA_KOKU_TIER_2 | 0.3 koku | Established House — significant for low-income samurai, routine for lords |
| GEISHA_KOKU_TIER_3 | 1.0 koku | Famous House — 1 full season of Local Daimyo stipend per visit; reserved for the elite |

---

### A4: Patron Visit Disposition Gain (§57.44.5)

PATRON_VISIT_DISPOSITION_GAIN = **+1 per visit** toward assigned geisha.

Calibrated against: bathhouse multi-character drift (+1 per shared evening, §57.44.8),
daily conversation gain (+1 per shared conversation, §12.6). Baseline 0 → Friend tier
(+31) requires 31 visits — roughly 1.4 seasons of nightly visits, or several years of
occasional stays.

---

### A5–A14: Geisha Routing Probability (geisha → okaasan) (§57.45.7)

Formula: `P = clamp(BASE + modifiers, MIN, MAX)`

Result is a float probability. Roll: `randf() < P` → route topic.

| Constant | Value | GDD Basis |
|---|---|---|
| GEISHA_BASE_ROUTE_CHANCE | 0.50 | Neutral starting point — §57.45.7 does not imply a directional default |
| GEISHA_OKAASAN_DISP_WEIGHT | 0.003 | Per point of disp_toward_okaasan: +0.3% per point = ±30% at max ±100 disp |
| GEISHA_PATRON_DISP_WEIGHT | 0.003 | Per point of disp_toward_patron: -0.3% per point (symmetric penalty) |
| GEISHA_CHUGI_BONUS | +0.15 | §57.45.7: "high Chugi (duty)" pushes toward routing |
| GEISHA_JIN_PENALTY | -0.15 | §57.45.7: "high Jin (compassion — does not want to see him harmed)" |
| GEISHA_SEIGYO_BONUS | +0.10 | §57.45.7: "high Seigyo (follows the system)" |
| GEISHA_ISHI_PENALTY | -0.15 | §57.45.7 / §19: Ishi virtue → withholds for personal leverage |
| GEISHA_TOPIC_TIER_PENALTY | -0.10 per severity step | §57.45.7: "topic severity (a Tier 1 secret … is harder to hand over)" |
| GEISHA_ROUTE_MIN | 0.05 | Floor — always some uncertainty even for a perfectly loyal geisha |
| GEISHA_ROUTE_MAX | 0.95 | Ceiling — always some uncertainty even for a perfectly disloyal geisha |

Severity steps above TIER_4: `3 - tier_enum_value` (enum TIER_1=0, TIER_2=1,
TIER_3=2, TIER_4=3). TIER_4 = 0 steps (no penalty), TIER_1 = 3 steps (-0.30).

Example outcomes at neutral disposition, no virtue modifiers:
- TIER_4 topic: 50% route chance
- TIER_3 topic: 40%
- TIER_2 topic: 30%
- TIER_1 topic: 20%

A loyal Chugi geisha (okaasan disp +50, no patron attachment) routing a TIER_1 topic:
0.50 + 0.15 + (50 × 0.003) = 0.50 + 0.15 + 0.15 − 0.30 = 50%.
A Jin geisha who has grown close to her patron (patron disp +60) routing TIER_1:
0.50 − 0.15 − 0.18 − 0.30 = −0.13 → clamped to 5%.

### A15–A22: Okaasan Routing Probability (okaasan → handler) (§57.45.8)

Formula: `P = clamp(BASE + modifiers, MIN, MAX)`

| Constant | Value | GDD Basis |
|---|---|---|
| OKAASAN_BASE_ROUTE_CHANCE | 0.65 | Business relationship with handler — okaasan usually cooperates |
| OKAASAN_HANDLER_DISP_WEIGHT | 0.004 | Slightly stronger than geisha-okaasan weight: handler relationship is more transactional |
| OKAASAN_CHUGI_BONUS | +0.10 | §57.45.8: "high Seigyo or Chugi" pushes toward routing |
| OKAASAN_SEIGYO_BONUS | +0.15 | §57.45.8: Seigyo is the primary reliable-operator virtue |
| OKAASAN_ISHI_PENALTY | -0.20 | §57.45.8: "high Ishi (resents being controlled)" — strongest withholding force |
| OKAASAN_TOPIC_TIER_BONUS | +0.10 per severity step | §57.45.8: "topic has clear political value (higher-tier topics are more worth reporting)" |
| OKAASAN_ROUTE_MIN | 0.05 | Floor |
| OKAASAN_ROUTE_MAX | 0.95 | Ceiling |

Independent okiya: okaasan routing roll does not fire (no handler to route to).
Topic stays in okaasan's known_topics for her own use.

### A23–A24: Kolat Eavesdropping (§57.45.9)

TN formula: `TN = KOLAT_BASE_TN + (okaasan.Stealth_rank × KOLAT_STEALTH_MULTIPLIER)`

Roll: Investigation (Notice) / Perception against TN. Success → topic enters agent's
topic_pool tagged with KnowledgeEntry (entry_type = "kolat_intelligence",
data = {"topic_id": id, "kolat_sourced": true}).

| Constant | Value | Rationale |
|---|---|---|
| KOLAT_BASE_TN | 10 | Baseline — an untrained okaasan has TN 10 (achievable by a competent agent) |
| KOLAT_STEALTH_MULTIPLIER | 5 | Per Stealth rank: +5 TN per rank (matches core L5R 4e Raise = +5 scale) |

Eavesdropping fires after the geisha routes to the okaasan (okaasan receives topic),
independently of whether okaasan then routes to the Scorpion handler.

### A32: Kolat Infiltration Rate (§57.45.14)

KOLAT_INFILTRATION_CHANCE = **0.15** (15% flat, all okiya regardless of control type).

§57.45.14: "each okiya has a flat 15% chance of containing a Kolat Silk agent at
world generation, regardless of handler type."

---

### A33: Imperial Capital Okiya Count (§57.45.2)

IMPERIAL_CAPITAL_OKIYA_COUNT = **3**

§57.45.2: "3–4 okiya of varying quality tier." Locked at 3: one Tier 1, one Tier 2,
one Tier 3. Reflects the GDD's floor of the 3–4 range and avoids over-clustering.

---

### A34: Okiya Tier Assignment by Settlement Type

Tier assignment uses settlement type and clan for cities.

| Settlement Type | Default Tier | Crane / Scorpion cities |
|---|---|---|
| CITY | 2 | 3 |
| FAMILY_CASTLE | 1 | 2 (Crane / Scorpion only) |
| CASTLE | 1 | 1 |

Imperial Capital (IMPERIAL_CAPITAL): three okiya at tiers 1, 2, and 3 respectively.
Non-castle / non-city types (VILLAGE, FORTIFICATION, KEEP, WALL_TOWER, TEMPLE,
SHINDEN, MONASTERY, TOWN): no okiya.

CASTLE settlements get an okiya with 70% probability (§57.45.3: "small or heavily
militarized castle towns may lack the zone entirely"). Probability constant:
CASTLE_OKIYA_CHANCE = 0.70.

### OkiyaData Resource Structure

```
okiya_id: int = -1                         # world-unique ID
settlement_id: String = ""                 # settlement this okiya belongs to
tier: int = 1                              # 1 / 2 / 3 per §57.45.3
is_scorpion_controlled: bool = false       # true = Bayushi handler pipeline active
okaasan_id: int = -1                       # NPC ID of the house mistress
geisha_ids: Array[int] = []               # NPC IDs of resident geisha
handler_id: int = -1                       # Bayushi handler NPC (-1 if Independent)
kolat_agent_id: int = -1                   # Kolat Silk embedded agent (-1 if none)
is_active: bool = true                     # false when okiya is closed/destroyed
```

Character sheet additions (§57.45.15):
```
assigned_geisha_ids: Dictionary = {}       # okiya_id (int) → geisha NPC id (int)
okiya_visit_counts: Dictionary = {}        # okiya_id (int) → visit count (int)
```

Settlement addition:
```
okiya_tier: int = 0                        # 0 = no okiya; 1/2/3 = tier
```

---

### No New ActionIDs

The geisha intelligence pipeline is entirely passive. NPCs visit via the Wind-Down
System (§57.44). Handlers maintain okaasan relationships through existing social
actions (DELIVER_GIFT, WRITE_LETTER, disposition visits). No new NeedTypes or
ActionIDs are introduced.
